# Usage

## Overview
Expo Router + React Native mobile starter with Andromo AI builder platform-managed Convex backend and Convex Auth.

- Frontend: Expo Router routes in `app/`
- UI: React Native primitives (`View`, `Text`, `Pressable`, `ScrollView`, `Image`)
- Backend: Convex files in `convex/` with auth tables preconfigured
- Client: `src/lib/convex.ts`
- Auth: Convex Auth with Password + Anonymous providers, email OTP verification, password reset, and SecureStore-backed React Native token storage
- Preview: `bun run dev` runs `convex dev --once`, starts an Andromo AI builder preview proxy, runs Expo web preview behind it, and exposes `/_expo/open?runtime=expo`
- Typecheck: `bun run typecheck`

## IMPORTANT: Starter Content
The existing app screen is only the temporary "Creating your app" placeholder shown while the app is being generated.

- Replace `app/index.tsx` with the requested app UI.
- Keep `app/_layout.tsx` wired with `ConvexAuthProvider`.
- Use the existing auth components when the app needs accounts; do not ask the user for a Convex project or auth provider.
- Add application tables to `convex/schema.ts` while preserving the pre-configured `...authTables`.
- Create queries, mutations, and actions in `convex/*.ts`; do not edit `convex/_generated/*`.

## Platform-Managed Convex
- This template is marked with `is_andromo_backend: true` in the template definition.
- Andromo AI builder provisions the Convex deployment and injects the deployment environment variables.
- Do not ask the user to log in to Convex.
- Do not ask the user to bring their own Convex project.
- Do not hard-code deployment URLs or dashboard keys.
- Use `EXPO_PUBLIC_CONVEX_URL` in mobile client code. The platform also supports the existing `VITE_CONVEX_URL` variable for compatibility.

## Code Organization

### Frontend Structure
- `app/_layout.tsx` - Expo Router stack and Convex Auth provider setup. Keep this provider when adding routes.
- `app/index.tsx` - Temporary generation placeholder; replace it with the requested app UI.
- `src/components/SignInForm.tsx` - Mobile auth form for sign up, sign in, email verification, forgot password, reset password, and guest sign-in.
- `src/components/SignOutButton.tsx` - Mobile sign-out control.
- `src/components/AuthGate.tsx` - Small helper that shows the auth form until the user is authenticated.
- `src/lib/convex.ts` - Convex client setup. Do not hard-code Convex URLs.
- `src/lib/auth-storage.ts` - Convex Auth token storage. Uses `expo-secure-store` on native and `localStorage` for web preview.
- `shared/auth.ts` - Shared password validation constants used by both the mobile form and Convex backend.

### Backend Structure (Convex)
- `convex/schema.ts` - Define application tables here; preserve `...authTables`.
- `convex/auth.ts` - Convex Auth setup with Password, email OTP verification/reset, and Anonymous providers. Do not modify unless explicitly changing auth behavior.
- `convex/auth.config.ts` - Convex Auth provider config. Keep it present for auth identity resolution.
- `convex/http.ts` - Registers Convex Auth HTTP routes. Keep this file when adding custom HTTP endpoints.
- `convex/router.ts` - Shared HTTP router for auth routes and any future custom routes.
- `convex/email/AndromoOTP.ts` - Platform-managed OTP email sender using Andromo SMTP.
- `convex/*.ts` - Add app-specific queries, mutations, and actions here.
- `convex/_generated/*` - Generated by Convex tooling; never edit or create manually.

## Mobile v1 Rules
- Keep the project Expo Go-compatible.
- Do not add Expo dev client setup, prebuild output, app-store submission, native binary build scripts, or custom native modules.
- Do not claim EAS, app-store, or native binary build support from inside the generated app. Andromo AI builder handles APK builds at the platform level.
- Use React Native primitives and Expo-compatible libraries.
- Do not use DOM elements such as `div`, `span`, `button`, `input`, or browser-only APIs in app screens.
- Do not copy web-only auth UI. Use `TextInput`, `Pressable`, `View`, `Text`, and the provided mobile auth components.
- Do not add browser-only storage for auth tokens. Keep `src/lib/auth-storage.ts` using SecureStore on native and web fallback only for preview.

## Authentication
Convex Auth is pre-configured with Password and Anonymous providers.

- Password auth includes email OTP verification and password reset flows.
- `src/components/SignInForm.tsx` handles sign up, sign in, email verification, forgot password, reset password, and anonymous sign-in.
- `src/components/AuthGate.tsx` can wrap authenticated app UI.
- `src/components/SignOutButton.tsx` signs out the current user.
- Email flows require `ANDROMO_SMTP_URL` and `ANDROMO_SMTP_API_KEY` in the Convex backend environment; Andromo AI builder injects these for platform-managed Convex projects.
- OTP email templates are `confirm-email` and `password-reset`.
- Do not remove `ConvexAuthProvider` from `app/_layout.tsx`; plain `ConvexProvider` will not send auth tokens.

Example mobile auth gate:

```tsx
import { Stack } from 'expo-router';
import { Text, View } from 'react-native';
import { AuthGate } from '@/components/AuthGate';
import { SignOutButton } from '@/components/SignOutButton';

export default function HomeScreen() {
	return (
		<>
			<Stack.Screen options={{ title: 'Home' }} />
			<AuthGate>
				<View>
					<SignOutButton />
					<Text>Authenticated app content</Text>
				</View>
			</AuthGate>
		</>
	);
}
```

## Convex Development
- Add app tables in `convex/schema.ts` after `...authTables`.
- Add queries, mutations, and actions in `convex/*.ts`.
- Use Convex validators for every function argument and return value where practical.
- Prefer indexed queries with bounded results.
- Use `getAuthUserId(ctx)` or `ctx.auth.getUserIdentity()` server-side for user-owned data.
- Never accept `userId` or any user identifier from the client for authorization.
- Do not edit `convex/_generated/*`; it is produced by Convex tooling.
- Run `bun run backend:deploy` after backend changes when the platform deployment needs updating.

Example authenticated Convex mutation:

```ts
import { getAuthUserId } from '@convex-dev/auth/server';
import { v } from 'convex/values';
import { mutation } from './_generated/server';

export const createItem = mutation({
	args: { title: v.string() },
	returns: v.id('items'),
	handler: async (ctx, args) => {
		const userId = await getAuthUserId(ctx);
		if (!userId) {
			throw new Error('Unauthenticated');
		}

		return await ctx.db.insert('items', {
			userId,
			title: args.title,
			createdAt: Date.now(),
		});
	},
});
```

## Auth Schema Rules
- `convex/schema.ts` already includes `...authTables` from `@convex-dev/auth/server`.
- `authTables` owns the Convex Auth `users` table and required auth indexes.
- Do not redefine the auth-owned `users` table for app profile data.
- If you explicitly customize the Auth `users` table, preserve Convex Auth compatibility and the framework-required `.index("email", ["email"])` name.
- For app profiles, create a separate table such as `profiles` with `userId: v.id("users")` and a `by_userId` index.

## Demo, Initial, and User-Scoped Data
When the developer asks you to add, seed, or change backend data, do it directly in Convex with a proper query/mutation/action or one-off script.

- First decide whether the requested data is global app data or belongs to a specific end user.
- For global app data, seed or update it directly in the appropriate Convex table.
- For end-user-scoped data, inspect existing backend `users` records before choosing a `userId`.
- If exactly one backend user exists, use that user's `_id` for the requested data.
- If multiple backend users exist, ask the developer which user should receive the data and include useful identifiers such as email, name, and `_id`.
- If no backend users exist, do not invent a target user. Ask whether to create a user, wait for sign-in, or make the data global instead.
- Do not weaken public mutations or remove auth checks just to seed data.
- For user-facing behavior, derive the user server-side from auth. Do not accept a `userId` argument for authorization.

## Convex Backend Rules
- Use the new Convex function syntax: `query({ args, returns, handler })`, `mutation({ args, returns, handler })`, and `action({ args, returns, handler })`.
- Always include argument validators for public and internal Convex functions.
- Add return validators where practical, especially for public app functions.
- Prefer indexes and bounded reads (`take`, `paginate`) over unbounded `.collect()`.
- Do not use `filter` for production queries when an index can express the access pattern.
- Do not store unbounded child lists inside a single document; create separate tables with foreign keys.
- Use `internalQuery`, `internalMutation`, and `internalAction` for private implementation details.
- Use `"use node";` only in files that export Node runtime actions. Never put `"use node";` in files that export queries or mutations.
- HTTP endpoints belong in `convex/http.ts` and use `httpAction`; keep `auth.addHttpRoutes(http)` registered.

## Routing
- Add screens under `app/`.
- Use `Stack.Screen` for titles and header options.
- Use `Link` from `expo-router` for navigation.
- Keep route files small and move reusable UI into `src/components/`.

## Verification
- Run `bun run typecheck` after code changes.
- Run `bun run dev` to preview on web and through Expo Go.
- Run `bun run backend:deploy` after backend changes when the platform deployment needs to receive updated Convex functions.
- Use the platform-facing `$PORT` URL for both the web iframe and `/_expo/open?runtime=expo`; the template only keeps that port stable while forwarding to Expo internally.
- Do not add app-level Expo manifest or bundle URL rewriting. Andromo AI builder handles remote Expo Go manifest URL rewriting at the platform proxy.
